# Lessons

- During upstream sync, conflict resolution must be a three-way integration. Preserve Tarot/Game ownership work while explicitly reviewing and carrying forward useful upstream deltas; `--ours` is only a temporary staging move, not a final merge strategy.
- When adapting a new Zetta Onyx preset, treat Handshake and NSFW/adult-mode blocks as first-class donor deltas. Even if the version diff looks focused on BOLT, explicitly compare `main`, `zettaHandshakeAsk`, `zettaHandshakeGrant`, and the active adult-mode block, then update prompt tests for their current wording.
- When the user corrects an observability diagnosis, first distinguish global logger debug from per-request debug. In this codebase, visible pasted prompt logs can come from `requestDebug` via `debugLog(...)` even when `logger.isLevelEnabled("debug")` is false.
- Tarot agents must have hard ownership boundaries. Justice owns adjudication and rolls, Emperor owns turn composition, Chariot owns HUD widgets, and Tower/GM must not remain a fallback command bus for mechanics that a Tarot agent owns.
- Do not present partial ownership work as complete. If the user asked for hard agent responsibilities, remove the fallback path instead of shipping a "first slice" that leaves the old responsibility leak intact.
- Do not confuse removing command output with fixing agent ownership. Tower's problem is not only that it can emit commands; it is that the current GM prompt feeds it the whole Game brain. Tower must be a narrative renderer with a narrow scene brief, not the owner of planning, adjudication, widgets, map/inventory decisions, or broad hidden campaign control.
- Do not stop after a "first architectural step" when the user asked for the whole task. If context is removed from Tower, immediately redistribute it to the responsible agents and verify the end-to-end ownership model before reporting done.
- When the user asks to test a user flow, preserve production-like chat/model parameters. Do not silently reduce max tokens, verbosity, reasoning, or other generation settings just to make the run cheaper or faster.
- Roll visibility in Game Mode means a first-class in-game UI card near the narration, not only an agent thought bubble or console/SSE evidence. The card must show what was checked, DC, roll, margin, and success/failure.
- Before counting a production-like flow as evidence, verify the exact requested model version in the active chat connection and the default agent connection. If the user says GLM 5.2, a GLM 5.1 run is only a discarded warm-up.
- For local UI verification, use Codex/headless browser tooling. Do not start app commands that auto-open the user's system browser; this repo's Vite config has `open: true`, so use the backend-served UI, headless Playwright, or an explicit no-open launch.
- When verifying transient UI like roll cards, assert the actual rendered text case-insensitively and capture evidence while the state is visible. If the card disappears during scene progression, treat that as a product bug and fix the lifecycle instead of accepting a fleeting SSE-only pass.
- When the user points at a SillyTavern donor preset as the standard, determine active blocks from `prompt_order`, because prompt objects can keep `enabled:false` while the order row enables them. For Zetta prose, Hermit must carry the active Onyx head prose bans, banned vocabulary, anti-echo, forward-motion, cinematic realism, and balanced adult routing instead of a generic "polish the text" prompt.
- When the user explicitly chooses an A/B donor block such as BOLT v2, treat that as the integration target even when the source preset currently keeps the block disabled in `prompt_order`; document the override and adapt its mechanism, not the old active fallback.
- Emperor command ownership includes command hygiene. Structured `commands` output must be normalized before save: malformed multiline tags, nested command tags, unsafe readable brackets, and single-string choice blobs are bugs at the Emperor boundary, not parser/UI problems to tolerate later.
- Distinguish upstream Marinara behavior from fork policy. If the original GM allowed player/internal thoughts, describe our stricter player-agency rule as a conscious TarotEngine contract, not as something upstream already prohibited.
- Do not patch GM prompt leaks with isolated negative bans. For long Game runs, define one visible-surface contract shared by system prompt, Tower prompt, BOLT audit, and output reminder so POV, player agency, NPC interiority, VN labels, and Tarot ownership cannot drift into conflicting instructions.
- Do not auto-advance the donor preset version just because a newer Template root file exists. If the user says to wait on a newer Zetta version, keep the explicitly approved archived preset as the source and treat the newer file as out of scope until approved.
- When live smoke still shows residual long-run failure, fix the harness or prompt contract before calling the flow ready. A partial or earlier clean run is only historical evidence.
- Hermit JSON prose can fail on long turns. The Game pipeline must retry on parse failure, then abort the turn before save if Hermit still fails; keeping original Tower prose as an accepted fallback leaks exactly the prose Hermit owns.
- Tower narrative-only stripping and segment-edit stripping are separate contracts. Segment editing may preserve readable Note/Book tags for parsers, while player-visible Tower prose must remove malformed/nested command and readable tags.
- Emperor player-facing command text must follow the Game setup language contract. If the setup language is English, choice labels and readable command copy from Emperor must be English even when the system prompt is Russian.
- Runtime sanitization and diagnostic events must expose the same contract. If Emperor commands are normalized before save, the `agent_result` SSE data must also contain normalized commands so UI/debug tooling does not present raw malformed commands as pipeline output.
- Hermit acceptance is not the final visible-surface gate. Re-run narrative-only sanitation after Hermit so accepted prose cannot reintroduce standalone italic thought lines, command tags, or readable artifacts into saved Game narration.
- Hermit owns prose sanitation. Phrase final runtime cleanup as contract enforcement/fallback only; the Hermit application path itself must normalize banned prose surface artifacts such as em dashes, including `changed=false` revisions.
- In Tarot-owned Game mode, Justice and Emperor are critical gates, not advisory helpers. If either fails, abort Tower generation visibly instead of letting Tower resolve mechanics or composition from a fallback prompt.
- Smoke harnesses must fail on SSE `error` and missing `message_saved`. Reading the last assistant message after a failed turn creates false-positive userflow evidence.
- SPA fallback must exclude built client asset paths. Missing `/assets/*.js` should never receive `index.html`; return JavaScript stale-build recovery or a real 404 so browsers do not crash with strict MIME `text/html` module errors.
- Visible prose cleanup must be enforced at the final save/stream boundary as well as inside Hermit. If a post-agent branch can return raw `trim()` output, banned typography such as em dashes can leak even when Hermit's own revision contract is correct.
- When a Game subsystem feels "interesting but cursed", identify its canonical owner before changing prompts. Bond/NPC state must have a server-owned source of truth; client-side parsing is presentation/support code, not the authority for reputation or NPC registry mutation.
- For Game UI/server contract changes, playtest against the actually running backend build on `:7860`. Passing src lint/tests is insufficient when the user is running `node dist/index.js`; rebuild/restart dist and exercise the real endpoint/UI flow.
- Never run smoke mutations against the user's live Game chat. For widget/session/gameplay smoke, create a temporary chat/branch or clone the campaign state, run the mutation there, and delete/ignore the test artifact after verification.
- Do not use regex/string surgery to "fix" prose style violations such as Zetta banned constructions. Prose cleanup belongs to Hermit with explicit donor-style instructions and banned constructions; runtime code may enforce mechanical safety, not rewrite literary meaning.
- When describing Tarot Game ownership, include Tower explicitly even though it runs through the main GM harness rather than the generic agent executor. Hidden harness-owned roles still belong in the visible chain contract.
- When a user shows a live prose leak, verify the actual agent log before attributing the failure to an upstream agent. Hermit failures require checking whether Hermit ran, whether it returned changed=false, whether its revision was accepted/rejected, and what final text was saved.
- Main Game reasoning does not prove agent reasoning. When enabling thinking for Tarot agents, verify the actual agent debug request fields (`reasoningEffort`, provider thinking toggle, reasoning tokens/thinking events when available) in a temp userflow.
